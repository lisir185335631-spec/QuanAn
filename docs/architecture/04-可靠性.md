# QuanAn 架构 · 承重墙④ 可靠性 / 失败处理

> 按《Agent 项目架构实质性标准》做到 **L3**。标准见 KB `2-standards/conventions/Agent项目架构实质性标准.md`。接地 `文件:行号`。

## 〇、一句定性

可靠性在 QuanAn = **在一个"必然会错"的 LLM 上,用多层兜底建出"用户永远看得到结果"的系统**。核心张力:LLM 输出概率性、不可信;但 9 步向导要求每步返回**确定性结构**(Zod schema)才能驱动下一步。**它是承重墙④**:去掉这层,LLM 任意抖动直接穿透成用户侧 500,对串行流程是致命的(`BaseSpecialist.ts:165-228`)。

## 一、运转机制(追踪两个真实失败场景)

**场景 A:step4 LLM 返回非法 schema**
`invokeLLM`(:106)→ `LLMGateway.complete`(`llm-gateway/index.ts:173`)返 content → `outputSchema.safeParse`(:116)失败 → **retry 1 次**再调 `invokeLLM`(:123)→ 仍失败 → `throw SchemaValidationError`(:132)→ 外层 catch(:165)→ 判 `isFallbackable=true`(非 TypeError/ReferenceError,:188)→ 查 `fallbackTemplate[mode]`:有则返静态模板(`isFallback=true`、cost_log model='fallback',:192-228),**无则 re-throw 真错给用户**(:195,AC-9)。

**场景 B:主模型超时**
`_callAnthropic` AbortController 在 `timeout_ms` 触发(:300)→ primary 失败 → `_callWithRetry(fallbackModel, 0)` 零 retry(:197)→ fallback 也挂 → **both-failed:写 `cost_log(success=false, errorCode='BOTH_FAILED')` + 不 throw,返硬编码中文串**(:204-214)→ 该串回到 Specialist、`safeParse` 必然失败 → 走场景 A 后半段 → fallbackTemplate 兜底。即:**both-failed 时用户最终看到的是 Specialist 的静态 fallbackTemplate,Gateway 那句中文串被 Zod 过滤掉了**。

## 二、设计理由(从本项目约束推导)

分层防护(外→内,共 6 层):L0 限流(Upstash token bucket,`rate-limiter.ts:80`)→ L1 网关 retry 1 → L2 fallback model → L3 both-failed 友好串(不 throw)→ L4 Specialist schema retry 1(:116-133)→ L5 fallbackTemplate(:192-228)→ L6 敏感行业 disclaimer 注入(:155)。

- **为什么 fallback 返模板而非抛错**:PRD 要求"用户不看空白页"——返有意义占位(如 PositioningAgent 的通用执行计划框架)保 UX 连续,`isFallback=true` 供运营排查,`logger.warn` 不静默隐藏。
- **为什么 both-failed 不 throw**:让 Specialist 的 fallbackTemplate 有机会兜底(throw 和返串最终都走 fallbackTemplate,但"不抛"在 SSE `stream()` 路径更安全 :233-235)。
- **为什么结构约束 + 还要 Zod**:Anthropic 用 tool_use 强制结构化(`anthropic-provider.ts:47-58`),但 `input_schema` 设 `additionalProperties: true`(宽松，不严格校验字段),**结构约束但非严格 schema**,故 Zod 校验 + retry 补刀保证字段名/类型正确。

## 三、失败边界(二阶失败:失败处理本身失败时)

- **fallbackTemplate 缺该 mode** → re-throw 真错(:195),用户看错误(可接受:新增 mode 未同步模板是开发期问题)。
- **降级链全挂(both-failed)** → 硬编码串被 Zod 过滤 → fallbackTemplate(若有)→ 正常降级响应;**VoiceChatAgent 例外**:override execute 不走此 catch、无 fallbackTemplate,失败直接 yield `{type:'error'}`(:165)。
- **cost_log 写失败** → Gateway 层(`cost-logger.ts:56-84`)与 Specialist 层(:285-310)均为 `await prisma.costLog.create` + `catch(err) { logger.error(...) }`——**await+catch logger.error，失败不阻断业务**(代价:成本数据可能丢,靠 traceId 日志对账)。
- **rate-limiter DB 故障** → `getUserQuota` 返 null → 默认 free 配额(:61-70);Upstash 未配置 → 限流直接 pass-through + warn(:83)。
- **ContextAssembler 抛错** → `assemble`(:95)无 try-catch → 传外层 catch → 走 fallback。

## 四、承重 / 软肋(软肋附"为什么接受")

| | 判定 | 说明 |
|---|---|---|
| 6 层兜底链 | **承重** | 用户可用性靠它;LLM 抖动不穿透 |
| schema retry + fallbackTemplate | **承重(真保障在 fallbackTemplate)** | retry 只赌"第二次格式对";**复杂 schema 两次都可能失败 → 真正保用户可用的是 fallbackTemplate,不是 retry** |
| **熔断器是注释非代码**(`index.ts:7` 写"5 次 5xx 熔断",全文件无 `CircuitBreaker`/`consecutiveErrors`) | **软肋** | 接受:当前规模有限流 + 双模型 + 友好串兜底,无雪崩;**大规模并发降级时会反复打爆 primary**——P1-P2 规模后必补 |
| **embed() stub 全零向量**(`index.ts:249-252`,TODO P3) | **软肋(牵连承重墙①/③ 的 L5)** | 接受:P0/P1 阶段 RAG 增强"形同未启用";**但需与承重墙③ L5 索引端一起核实**——若 chunk 向量真由此 stub 生成,RAG 召回无意义 |

## 五、根本张力与关键赌注

- **赌注:schema retry 赌一次就够**(总 2 次 LLM 调用)。赌"同 prompt 第二次大概率格式对"。复杂/深嵌套 schema 两次都败 → 必走 fallbackTemplate → **fallbackTemplate 覆盖度才是真正的可用性底线**。
- **赌注:fallbackTemplate 静态内容够有意义**(通用而非个性化,用户感知"系统繁忙,以下为通用备用方案"=可接受降级)。
- **赌注:双模型不同时挂**。primary+fallback 跨 tier,但**同 provider**——Anthropic 区域故障 → 两者皆挂 → both-failed 必触发。
- **赌注:cost_log await+catch 不阻断业务、失败只记日志的可观测性够用**(靠 traceId 日志;长期 DB 挂 → 成本盲区)。
