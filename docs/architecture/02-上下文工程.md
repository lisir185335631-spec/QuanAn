# QuanAn 架构 · 承重墙① 上下文工程

> 按《Agent 项目架构实质性标准》做到 **L3**。标准见 KB `2-standards/conventions/Agent项目架构实质性标准.md`。每个论断接地 `文件:行号`。

## 〇、一句定性

上下文工程在 QuanAn = **ContextAssembler 7 路并行装配 + 全量注入 + 容错降级**。这是"认知引擎吃什么"的供养层——它决定每个 Specialist 在每一步看到什么、从而产出什么。**它是承重墙①**:拆掉它,所有 Specialist 收到的是空白通用 prompt,退化成无状态 LLM wrapper,系统不再是 Agent(`ContextAssembler.ts:53-129`)。

## 一、运转机制(追踪 PositioningAgent 执行 step4)

`BaseSpecialist.execute`(`specialists/base/BaseSpecialist.ts:82`)第 2 步无条件调 `contextAssembler.assemble`(:95),后者 7 路 `Promise.allSettled` 并行(`ContextAssembler.ts:54-63`),各路 `withTimeout(5000ms)`(:31,56-62),理论总耗时 ≤800ms:

| 路 | 取什么 | 源 |
|---|---|---|
| 1 L2 | 该账户**全量** stepData(无 LIMIT,createdAt asc) | `_fetchStepData` :152-156 |
| 2 L4 | `evolutionProfile.latestInsight`(画像) | `l4-profile.ts:13-23` |
| 3 样本 | **stub 恒空** | :161-163 |
| 4 L5 | RAG pgvector(按 agentId 路由 topK) ⚠️ | `ContextAssembler._fetchRag` :169-210(agentId 路由逻辑在此);`retrieve.ts:75-83` 是 pgvector 查询实现(不感知 agentId)；**索引端 `embed()` 全零 stub（见 03-记忆/04-可靠性），召回退化存疑** |
| 5 常量 | 方法论常量(纯内存,0 IO,永不超时) | :212-214 |
| 6 prompt | DB prompt canary 版本 | :138-149 |
| 7 常量 | DB 常量 canary | :223-260 |

**systemPrompt 拼接顺序**(`_composeSystemPrompt` :297-371):persona(DB canary > `templates/positioning.ts`)→ `# 历史 step 摘要(L2)`(全量 stepData result JSON)→ `[Section 4] 用户偏好画像`(L4)→ `# 方法论(常量)`→ `[Section 6] RAG 参考`(各 chunk 截 200 字)→ `[Section 7] DB 常量`。**userPrompt** = `piiMask(userInput)` 包进 `<user_input>`(:374-379,`pii-mask.ts:44-68`)。step4 → PositioningAgent(execution)→ `Step4BaseSchema` → `llmGateway.complete(reasoning, 60s)`。

## 二、设计理由(从本项目约束推导)

- **7 路 allSettled 容错**:任一路超时/报错不 throw,`assemble` 总返回完整 ctx(最差全段降级为空)——因为 QuanAn 是面向非技术博主的同步向导,**任何一步都必须出结果、不能因某路 DB 抖动而整步失败**。
- **全量注入 stepData(无裁剪)**:`_fetchStepData` 无 LIMIT——**赌"账户 stepData 条目有界"**(9 步线性、每步覆写一次、典型 ≤10 行),全量注入 ≈ 完整 L2 上下文,省掉检索规划层。换个高频写入的项目这条就崩。
- **canary 走 DB 灰度 + 确定性哈希**:`md5(userId:specialistId)%100` 分桶(`prompt-version.service.ts:269-274`),同用户永远同版本,prompt 改版不发代码、可 1/10/50% 灰度。

## 三、失败边界(闭合到实际路径)

- **某路 5s 超时** → `allSettled` 捕获为 rejected → 对应变量置 null/[] → `layersUsed` 不含该层 → 拼接时该 Section **整段跳过,正常返回不 throw**。L2 超时 → 注入 `[新用户·暂无 step 数据]`(:315-317);L4 超时 → Section4 缺;L5 超时 → Section6 缺;DB prompt 超时 → 回落 `templates/*.ts`(:310-313)。
- **token 超限** → `contextTokens=chars/4`(:120)**只算不裁**;systemPrompt 超长直接传 LLM → 撞模型 context 硬限时 `llmGateway.complete` throw → 被 `BaseSpecialist` catch → 走 fallbackTemplate(**但 fallback 不含 L2/L4 数据,质量骤降**)。**ContextAssembler 层无任何保护截断**。

## 四、承重 / 软肋(软肋附"为什么接受")

| | 判定 | 说明 |
|---|---|---|
| 7 路 allSettled 降级 | **承重** | 系统可用性靠它;单路故障不传播 |
| 常量路(纯内存) | **承重** | 0 IO 永远成功 |
| 无 token 裁剪 | **软肋** | 接受:线性 9 步假设 stepData 有界、实现简单成本低 → **赌注 A,量一大就是成本/截断悬崖** |
| 样本路 stub(:161-163) | **声明,非承重** | 接受:PRD-8 明确延期、有意跑空,非遗漏 |
| `needRag` 字段定义但 `_fetchRag` 不读(`types.ts:19` vs `ContextAssembler.ts:169-210`,agentId 硬编码路由) | **软肋(形同虚设)** | 接受:当前 agentId 路由够用;字段是未兑现的扩展点 |
| PrivateDomainAgent 绕过 ctx(`:165-188` 收 `_ctx` 不用,自建 prompt) | **真缺陷** | 不接受:① 7 路白跑浪费 DB ② **`_buildUserPrompt`(:220-228)绕过 piiMask → 用户手机/邮箱原文进 LLM = LD-018 R-14 真漏洞** → 阶段6 修 |

## 五、根本张力与关键赌注

**赌注 A:全量注入 > 检索**。赌对:省检索规划层(无 query rewriter/reranker)、延迟可控、Specialist 不需声明"我要哪些数据"。赌错两条:① **量增长** → systemPrompt 膨胀 → 超 context → fallback(低质降级);② **信噪比** → RAG `retrieve` 返 topK 全量、similarity 只展示不筛选(`_buildSection6` :285-295)→ 低相关 chunk 也进 prompt 引入噪音。这是上下文工程最该补的一根弦(token budget + 相关性阈值)。
