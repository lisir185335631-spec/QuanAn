## 〇、一句定性

QuanAn 的编排**不是单一编排**，是 **三层编排 + 通过共享记忆层松耦合**：

```
① 同步主向导（9 步线性流水线）  ── registry 声明式派单，tRPC 阻塞，前端驱动
② 按需工具（主向导外的 Specialist）── 独立 route，用户主动触发（分析/私域/诊断…）
③ 异步自治（BullMQ + cron）      ── Evolution 反馈飞轮 / DailyTask / 知识摄取
        ↕ 三层互不直接调用，靠【共享记忆层】耦合 ↕
   ContextAssembler(读) ←──→ stepData / evolutionProfile / pgvector(写)
```

这是 QuanAn 作为 Agent 系统的**承重骨架**：去掉①系统不再"引导孵化"，去掉③系统永久静态（不学习），去掉中间的共享记忆层三层就退化成各自孤立的 LLM 包装器。

## 一、业务锚（L3-2 的前提：编排服务于哪条真实业务流）

**主业务流**：用户从"选行业 + 传资料"出发，经 9 步向导，产出一套可运营的 IP 资产（定位→人设→执行计划→变现→选题→拍摄→文案→直播）。这是一条**"专家方法论驱动的 IP 孵化流水线"**。

**真实 9 步**（权威定义 `apps/api/src/lib/constants/steps.ts`，运行时断言 `STEPS.length !== 9` @L46；**刻意无 step2**，L5 注释"实测原版 404 · spec §ⅩⅦ"）：

| 顺序 | stepKey | 业务语义 | Agent(mode) | 业务层 |
|---|---|---|---|---|
| 1 | step1 | 行业选择 | PositioningAgent(industry) | 定位 |
| 2 | step3 | 账号包装 | BrandingAgent(packaging) | 定位 |
| 3 | step3b | 人设定制 | BrandingAgent(persona) | 定位 |
| 4 | step4 | 执行计划 | PositioningAgent(execution) | 规划 |
| 5 | step4b | 变现路径 | MonetizationAgent(ladder) | 规划 |
| 6 | step5 | 爆款选题 | TopicAgent(category) | 内容生产 |
| 7 | step6 | 拍摄计划 | VideoAgent(shooting) | 内容生产 |
| 8 | step7 | 文案生成 | CopywritingAgent | 内容生产 |
| 9 | step8 | 直播策划 | LivestreamAgent | 内容生产 |

> 注：14 个能力工具里只有 6 个 Specialist 在主向导上（Positioning/Branding/Monetization/Topic/Video/Copywriting/Livestream）；Analysis/PrivateDomain/Diagnosis/DeepLearn 等是②按需工具，走独立 route 不在 9 步链上。

## 二、运转机制（L3-1：追踪一个真实请求）

### 单步同构时序

每一步同构（触发点 `apps/api/src/trpc/routers/app/stepData.ts:110` → `BaseSpecialist.execute` `apps/api/src/specialists/base/BaseSpecialist.ts:82`）：

```
用户在 /step/X 填表/点生成
  → tRPC stepData.save({stepKey, inputs})            [stepData.ts]
  → findStepAgent(stepKey)                            [registry.ts:173] 顺序匹配 STEP_AGENT_REGISTRY(:103-170)，首中即停
  → entry.agent.execute(req)                          [BaseSpecialist.ts:82] 4 步模板：
        1. inputSchema.parse(userInput)               (:92)  入参校验
        2. ContextAssembler.assemble({accountId,...}) (:95)  7 路并行组装上下文（见下）
        3. invokeLLM(ctx, req)                         (:106) 抽象方法·子类唯一 LLM 钩子·强制单次不循环(:256)
        4. outputSchema.safeParse(content)            (:116-133) 失败 retry 1 次 → 仍失败抛 SchemaValidationError
  → persistStepAgentResult()                          [registry.ts:181-202] upsert step_data 表
  → entry.afterPersist?.()                            [registry.ts:150-161] 如 step7 额外写 history 表
  → 前端渲染（step5/step7 走 SSE saveStream :165，其余同步返回）
```

全程阻塞在单个 Postgres 事务内（`stepData.save` `$transaction` timeout=70000ms，容纳 LLM 最长 ~45s）。

### 追踪一个真实请求：step4「执行计划」

step4 是最能暴露编排命门的一步——它的产出依赖 step1（行业）和 step3（账号包装）：

1. 用户在 `/step/4`，前端 `useStepData` 已从 localStorage 载入 step1/step3 产出；页面调 `readOtherStep(accountId,'step1')` `apps/web/src/hooks/useStepData.ts:25-38` 读 `step1.industry` 预填（`Step4.tsx:309`）。
2. 点生成 → `stepData.save({stepKey:'step4', inputs})`。
3. `findStepAgent('step4')` 命中 **PositioningAgent(execution mode)**。
4. `BaseSpecialist.execute`：
   - `ContextAssembler.assemble` 7 路并行，**路1 `_fetchStepData(accountId)` 用 `prisma.stepData.findMany({where:{accountId}})` 把该账户已完成的 step1/step3/… 全量捞出**，序列化进 systemPrompt 的"历史 step 摘要(L2)"段（`ContextAssembler.ts:317-321`）。
   - `invokeLLM` → `LLMGateway.complete(model_tier='reasoning')`（PositioningAgent 配置 reasoning，`POSITIONING_CONFIG:99`）→ Anthropic。
   - `safeParse` 校验 `Step4BaseSchema` → 写回 `step_data`。

> **这里是编排的命门（L3 核心洞察）**：step4 拿到 step1/step3 的产出，**不是 step3→step4 的接口传参，而是 step4 执行时把全账户 stepData 整体捞进自己的上下文**。也就是 ——**编排的"数据流"实际跑在"共享上下文注入"里，不在"步骤接口"里**。这一条决定了下面所有的优劣与赌注。

### 三层如何松耦合（控制权与数据如何流转）

- **① 主向导**：控制权在**前端**——用户点击 `navigate('/step/X)` 决定下一步，**后端无状态机、无"当前步骤"字段、无顺序强制**（`step_data` 表只有 `status: completed/fallback`，无 `in_progress`）。跳步/回退/重做完全自由（`stepData.save` 是 upsert，接受任意 stepKey，`version` 自增计重做）。
- **③ 异步飞轮**：`feedback → 阈值 {5,20,50,100} 触发`（`lib/evolution/trigger.ts:20-54`，原子 SQL upsert + jobId 去重）`→ BullMQ evolutionQueue → EvolutionAgent` 重算画像 `_mergeInsight(:294-319)` → 原子事务写 `evolutionProfile.latestInsight`（`_writeEvolutionTransaction:322-387`）。**主向导从不调用飞轮**；飞轮的产出在下一次任意 Specialist 执行时经 `ContextAssembler` 路2 `getLatestInsight`（`l4-profile.ts:13-23`）读入、注入 systemPrompt Section4 —— **这就是"通过共享状态松耦合"的实证：单向反哺，不直接调用。**
- **ContextAssembler 7 路**（`ContextAssembler.ts:54-63`，各路 `withTimeout 5000ms` + `Promise.allSettled` 容错）：路1 L2 历史步骤 / 路2 L4 进化画像 / 路3 样本(**stub 恒空**) / 路4 L5 RAG(pgvector) / 路5 方法论常量 / 路6 DB prompt 版本(canary) / 路7 DB 常量(canary)。

## 三、设计理由（L3-2：从本项目约束推导，非通用理由）

| 设计 | 推导链（扣 QuanAn 画像/业务，换个项目就不成立） |
|---|---|
| **9 步固定线性**，非 Agent 自主规划 | IP 孵化是**有专家方法论的、有序确定**的流程（定位→包装→规划→内容），不存在"运行时才知道下一步做什么"的开放性 → 不需要 agency 去规划路径；固定主线还让用户能看懂每步在干嘛（P2 创作容错、面向非技术博主）。**对比**：一个开放式研究/编码 Agent 就必须给 agency，因为路径不可预定。 |
| **registry 声明式静态派单**，非 Supervisor/动态调度 | step→agent 是**编译期就确定**的静态映射（`registry.ts:103-170`），没有"运行时智能选 Agent"的需求 → 上 Supervisor 是过度设计。 |
| **每步单次 LLM、禁自循环**（`invokeLLM` 抽象单次 `:256`） | 每步是**确定性产出任务**（生成一份定位/一套文案），不是开放探索 → 不需要 ReAct/自循环；契合 LD-001（95% Workflow + 5% Agent）。赌"单次够用"——见赌注 E。 |
| **三层靠共享记忆层松耦合**，不直接互调 | 飞轮/每日任务是**后台学习**，绝不能阻塞用户的同步向导体验 → 必须异步且解耦；符合 LD-003（Specialist 不互调）。 |
| **跨步靠"全量上下文注入"**而非步骤接口传参 | 9 步产出是**弱结构、强语义**的文本（定位话术/人设描述），下游 Specialist 需要的是"语境"而非"几个字段" → 全量注入比定义 N×N 个步骤间接口更省事。**这是一个赌注**（赌注 A），不是免费的——见五、六。 |

## 四、失败边界（L3-3：闭合到实际路径，不只指出"有问题"）

| 失败点 | 系统实际怎么走（接地） |
|---|---|
| LLM 输出 schema 不合 | `BaseSpecialist.execute:116-133` retry 1 次；二次仍失败 → 抛 `SchemaValidationError`，用户侧看到错误（**不静默**） |
| LLM 超时 | `:108-113` `AbortError` → `LLMTimeoutError`（事务 70s 上限内） |
| 可降级错误 | `:165-228` 若 Specialist 有 `fallbackTemplate[mode]` → 返回静态模板，`isFallback=true`、`status='fallback'` 写库；**无模板则透传真错**（:195 注释"让用户看真错"） |
| 主模型挂 | `LLMGateway.complete:189-215` 降级到 fallback model；两者都挂 → 返回**硬编码兜底字符串**（不 throw），写 `cost_log(success=false)` |
| 长流程中断（关浏览器） | 支持续跑：LS + DB 每步独立持久化，无"流程会话"概念，回来任意步 `useStepData.load()` 从 LS、`stepData.get` 从 DB 取最新态 |
| 跨账户串扰 | **双轨守门**（见条件承重墙⑥）——`ContextAssembler._fetchStepData`(`ContextAssembler.ts:151-156`) 自带 `where: { accountId }`（应用层过滤），同时 router 路径靠 `account-isolation.ts:56-64` 的 `SET LOCAL ROLE quanan_app + set_config` 走 RLS。两者并行：`_fetchStepData` 跑在模块级 prisma（脱事务 RLS）、靠 `where:{accountId}` 应用层守；router `stepData.ts:44-45` 则依赖 RLS（注释"RLS handles it"）。RLS 失效 = router 路径跨租户泄漏；`_fetchStepData` 靠应用层 where 仍有保护。详见承重墙⑥。 |

**缺口（声明了但不是实现）**：`llm-gateway/index.ts:7` 头注释写"熔断（同模型连续 5 次 5xx）"，但全文件无 `CircuitBreaker`/`consecutiveErrors` 任何状态机——**熔断器是注释，不是代码**。实际只有"主→retry→备→兜底串"，无熔断。

## 五、根本张力与关键赌注（L3-5）

这套编排架构押了 5 个赌注，每个都省了东西，也都埋了悬崖：

| 赌注 | 赌对省了 | 赌错会怎样（接地） |
|---|---|---|
| **A. 9 步固定 + 全账户 stepData 全量注入够用** | 免去相关性排序、token budget 管理、动态记忆检索 | **最近的悬崖**：step8 要消化 step1-7 全部 JSON，`ContextAssembler.ts:120` `contextTokens` 只算不裁，context 利用率极低、成本线性涨、最终撞模型硬限——**当前零防护代码** |
| **B. accountId 是租户边界，userId 在会话内无关** | 更简单的数据模型，Agent 面向账户 | 所有 Specialist 传 `userId=0` → `rate-limiter.ts:80-103` per-user 限流形同虚设；IP 团队多人场景无法做用户级配额/行为分析 |
| **C. localStorage 跨步预填足够安全** | 零 RTT 的 UI 流畅 | 无痕/清缓存/换设备 → `readOtherStep` 返回 null → 表单**静默回落默认值**（如 `?? '美业'`），用户以为拿到个性化建议、实际 LLM 收到硬编码默认——静默错误无提示 |
| **D. 反馈阈值 {5,20,50,100} 足够驱动演进** | 原子 SQL 无竞态、BullMQ 负载可预测 | 给 <5 条反馈的用户永不触发演进；**>100 条后永不再演进**——高频用户被冻结在最后一次画像 |
| **E. 单次 LLM 对所有步骤够用** | 无 agent loop、延迟可预期 | VideoAgent 分镜等复杂任务，质量上限被单次 prompt 锁死 |

**一句话张力**：QuanAn 用"**固定流程 + 全量上下文 + 单次调用**"换来了**极强的可控性与可预测性**（这正是它该要的，因为 IP 孵化是方法论驱动的），代价是**上下文经济性、个性化连续性、复杂任务质量**——这三者在用户量/使用深度增长时会先后触壁。

## 六、承重 / 软肋判断（L3-4：软肋附"为什么接受"）

**真承重墙**（去掉/塌了系统不成立）：① ContextAssembler 7 路（差异化行为的唯一来源）② registry 派单（9 步协调骨骼）④ BaseSpecialist 失败兜底链 ⑤ Evolution 飞轮闭环 ⑥ RLS 13 表隔离（`manual_rls.sql:49-98`，Agent 行为语境正确性的隐性前提）。

**软肋及为什么（暂时）接受**：

| 软肋 | 为什么接受 | 何时必须还债 |
|---|---|---|
| 跨步非类型安全（LS 字符串键 + prompt 全量注入） | 选了"共享记忆层"换 UI 流畅与省接口，弱结构文本本就难定强类型接口 | 步骤数/字段增长到改一步频繁碰碎别处时 |
| 无步骤依赖/前置校验，可跳步 | 创作流程要容错、允许用户非线性探索（P2） | 出现"跳步导致上下文缺失→产出明显降质"投诉时，需加软提示（非硬挡） |
| ContextAssembler 无 token 裁剪 | MVP 期账户 stepData 量小，未触壁 | **赌注 A 的悬崖**——用户完成步骤多了必须做，优先级最高 |
| L3 history 只写不读、L5 trending stub、样本路 stub | 记忆层分期落地（PRD-8/9） | 标注为**声明态非承重**，不得在架构图当承重墙画；对应 PRD 实装时补 |
| PrivateDomainAgent 忽略 ctx（`:165` 收 `_ctx` 不用，自建 prompt） | 无正当理由 = **真缺陷**，违 LD-007 | 阶段6 修（已在 audit-redlines R-11 抓到） |
| 前端 `stepConfig.ts` 另一套 step 定义死代码 | 无 = 维护陷阱 | 阶段6 清 |

## 七、L3 自检（对照《标准》六条 + 短板法）

| 承重墙（编排维度涉及） | 达标级 | 依据 |
|---|---|---|
| ② 控制流/agency 边界（**本维主轴**） | **L3** | 三层编排 + 单步时序 + step4 真实请求追踪 + 前端驱动控制流 + 设计理由扣画像 + 失败路径闭合 + 赌注 + 软肋附因 + 全程 file:line |
| ① 上下文工程（编排牵涉） | L2→L3 | 7 路机制 + 跨步注入真相已到 L3 的运转层；token 经济性赌注已点；**完整 L3 留承重墙① 专篇** |
| ③ 记忆/状态（编排牵涉） | L2 | 飞轮闭环 + 活跃/声明记忆已辨明；**完整 L3 留承重墙③ 专篇** |
| ④ 可靠性/失败 | L3（编排相关面） | 失败边界表闭合 + 熔断器"注释非实现"已揭 |
| ⑥ RLS 隔离（条件承重墙） | L2→L3 | 已点明"Agent 拿到别人的大脑"风险 + 接地；**完整 L3 留专篇** |

**结论**：编排维度（承重墙②）本篇达 **L3**。①③⑥ 在编排视角内已到 L2-L3 的相关面，其**完整 L3** 作为各自承重墙专篇（02 上下文工程 / 03 记忆 / 06 多租户隔离）继续做——这是按《标准》短板法"逐墙到 L3"的推进路线，非本篇遗漏。
