# QuanAn 架构 · 承重墙⑤ 演进 / 反馈

> 按《Agent 项目架构实质性标准》做到 **L3**。标准见 KB `2-standards/conventions/Agent项目架构实质性标准.md`。接地 `文件:行号`。

## 〇、一句定性

演进在 QuanAn = **反馈飞轮:从真实使用的 feedback 中学习、重算用户画像、反哺回每个 Specialist 的上下文**。这是"系统怎么持续变好"的承重墙⑤。但它的真实形态是 **"单向写入 + 人工兜底",不是数据驱动的自适应闭环**——这是本篇最关键的 L3 判断。

## 一、运转机制(追踪一次真实反馈如何演进)

`evolve` mutation(`evolution.ts:108-135`)→ ① `feedbackLog.create` 写 `feedback_logs` ② `void enqueueIfThresholdMet`(**异步不阻塞** mutation,AC-8)→ `trigger.ts:20-55`:原子 SQL `INSERT...ON CONFLICT DO UPDATE feedback_count_total+1 RETURNING`(防 race,:25-32),count ∈ `{5,20,50,100}`(:13)才 `evolutionQueue.add(jobId:'evo:{acc}:{count}')`(去重,:41-48)→ BullMQ worker(concurrency=5 在 `worker.ts:41`,attempts=3 退避在 `queue.ts:21`)→ EvolutionAgent:`feedbackN=max(1,floor(count/5))`(:234,**首触发 count=5 → 只取 1 条反馈**)+ `getDeepLearningSamples(10)` → `llmGateway.complete(reasoning, json_schema)` → `_mergeInsight`(fresh 优先、union 去重、截 top10,:290-319)→ `$transaction([profile.update, insight.create])`(:355-381)→ 下次任意 Specialist 经 `ContextAssembler` 路2 读 `latestInsight`、注入 `[Section 4]`(:330-335)。**单向反哺,主向导从不调飞轮**。

## 二、设计理由(从本项目约束推导)

- **阈值 {5,20,50,100} 递增**:对应等级语义(`lib/constants/evolution.ts:19-29`)L2(5)可手动进化→L3(20)自动→L4(50)选方向→L5(100)档案稳定可克隆;递增体现"边际信号递减"——早期少量反馈信号强,后期需更多才有新信号,避免低质频繁触发。
- **单向反哺(不让主向导直接调飞轮)**:演进是异步旁路(BullMQ),主链路只**只读**消费 latestInsight,防循环依赖与延迟注入。
- **prompt canary 灰度**:`canaryPct ∈ {0,1,10,50,100}`、确定性 `hash(userId:specialistId)%100`(`prompt-version.service.ts:268-275`)、100% 需 dual-approval、回滚需 super_admin+reason≥20字(canary rollback 约束见 `prompts.ts:248`「US-008 AC-7: canary rollback with reason ≥ 20 chars」注释；另有 prompt 版本回滚 `rollbackVersion:175`)。

## 三、失败边界(闭合到实际路径)

- **evolution job 失败** → BullMQ retry≤3 → `worker.on('failed')` attemptsMade≥3 写 `dead_letter_alert` 日志(`worker.ts:52-66`);**但 TODO P3 未接 PagerDuty/Sentry,无真死信队列**(仅 `removeOnFail:{count:50}`)。
- **`_mergeInsight` 冲突**(fresh 的 preferredCatchphrases 可能撞 prev 的 avoidList)→ **合并层无 hard-conflict 检测** → 由下一轮 `detectEvolutionAnomalies` 的 `conflicting_insights` 异步捕获、发 DingTalk 告警、不阻断写入(`anomaly-detection.service.ts:204-276`)。
- **LLM timeout / schema 二次失败** → 有 previousInsight 则返 `_buildFallbackResponse(prev)`、`isFallback=true`、**不写新 insight**(:141-148,168-175);无 prev 则抛。

## 四、承重 / 软肋(软肋附"为什么接受")

| | 判定 | 说明 |
|---|---|---|
| feedback→阈值→BullMQ→画像→反哺 闭环 | **承重** | 系统不学习则每用户永远同质量;此链路代码完整、有 retry/fallback/canary |
| **100 条后冻结**(count>100 永不再触发) | **刻意设计 + 软肋** | 接受:L5"档案稳定·走 clone 不再演进"语义;**但无自动解冻,`forceRebuildEvolution` 不重置 count**(`evolution-rebuild.service.ts:34-91`)→ 高频用户被永久冻结在最后画像 |
| PrivateDomainAgent 退出飞轮 | **刻意隔离** | 接受:私域话术风格固定不漂移;用户对私域输出的评价仍进飞轮,只是该 agent 本体不消费 |
| L3 history 不进飞轮 | **合理排除** | history 是输出记录非反馈信号;feedbackLog 才有 good/bad 标签 |
| **canary 无自动止损**(`ab-stop-loss.job` 存在但未连 canary 回滚) | **软肋** | 接受:有手动回滚 + dual-approval;规模后需补数据驱动止损 |
| **`evaluatePromptVersion` isMock=true**(`prompts.ts:152`)+ feedbackTrend/moduleRanking 返 mock(`evolution.ts:193-206`) | **软肋(效果度量空转)** | 接受:P0 先打通写入链路;**但意味着"演进有没有让系统变好"当前无法度量** |

## 五、根本张力与关键赌注

- **赌注:4 次触发够驱动演进**。首触发 feedbackN=1(1 条反馈就重算画像,信号质量存疑);union 累积**无淘汰机制**,早期低质 insight 会持续叠加;100 后冻结。
- **赌注:canary 够安全**。机制(确定性哈希+5 步灰度+dual-approval)健全,**但无数据驱动自动回滚**,`evaluatePromptVersion` isMock → LLM Judge 没真跑。
- **核心张力**:`satisfactionRate` 被算出来写进 profile(:352),**却未用于任何自动决策**。飞轮本质 = **单向写入 + 人工看板兜底**,不是"度量→自动调整"的自适应闭环。这是承重墙⑤ 离"真正会自己变好"最远的一段,也是它最大的开放性风险。
