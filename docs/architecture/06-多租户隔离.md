# QuanAn 架构 · 条件承重墙⑥ 多租户 / RLS 隔离

> 按《Agent 项目架构实质性标准》做到 **L3**。条件承重墙(QuanAn 多租户 → 命中)。标准见 KB `2-standards/conventions/Agent项目架构实质性标准.md`。接地 `文件:行号`。

## 〇、一句定性

多租户隔离在 QuanAn = **三道闸把租户隔离全压在 DB 行级安全(RLS)上**。**它对 Agent 项目是承重墙、不是普通"传统软件层"**——RLS 失效在这里不是"拿到不该拿的数据行",而是 **`ContextAssembler._fetchStepData` 跨账户返回 → LLM 把别账户的 IP 孵化历史(定位/人设/话题)当上下文 → Agent 以错误账户的"大脑"生成内容**。这是 Agent 特有失效模式。

## 一、运转机制(追踪一次请求的三道闸)

```
HTTP 请求
→ 闸1 createContext() 读 Lucia session → ctx.activeAccountId        (context.ts:74，信源=DB users.active_account_id，非前端传参)
→ protectedProcedure → 闸2 accountIsolationMiddleware              (account-isolation.ts:19-66)
    ctx.prisma.$transaction(async tx => {
      tx.$executeRaw`SET LOCAL ROLE quanan_app`          降权(超级用户默认 bypass RLS)
      tx`SELECT set_config('app.current_account_id', ${id}, true)`   事务级参数
      next({ ctx: { ...ctx, prisma: tx } })              把事务客户端作为后续 ctx.prisma
    })
→ resolver 用 ctx.prisma(=tx) 查 stepData.findMany()
→ 闸3 PG RLS policy: account_id = NULLIF(current_setting('app.current_account_id',true),'')::int  自动过滤  (manual_rls.sql:49-99)
```

`stepData.ts:44-45` 注释 `// ✅ No where:{accountId} — RLS handles it` 即此机制的依据。SSE 订阅特殊:用 `is_local=false`(连接级,因 `$transaction` 生成器返回即提交)。

## 二、设计理由(从本项目约束推导)

- **委托 DB RLS 而非应用层 `where:{accountId}`**:应用层 where 分散在每个 router,**漏一处即跨租户泄漏、且无中心守门**;DB RLS 是 kernel 级 enforce、不可绕(除超级用户)、单一控制点(`account-isolation.ts` 是 `apps/api/src` 内唯一 `$executeRaw` 处,LD-009 R-009)。
- **SET LOCAL ROLE + set_config 双重缺一不可**:只降权不 set_config → policy 里 `current_setting` 返 NULL → 全表 0 行(default-deny);只 set_config 不降权 → 超级用户 bypass RLS → 全表可见。

## 三、失败边界(用 trending_favorites 漏 RLS 做实例)

`trending_favorites`(`schema.prisma:573-586`)注释明写"per-account RLS 保护"、有 `accountId` 字段+外键,但 **`manual_rls.sql` 中无任何一行**(grep 为空),建表 migration 也无 `ENABLE RLS`。实际后果链:
1. 无 RLS policy → PG 默认 PERMISSIVE(无限制)。
2. 当前所有 caller(`trending.ts:185,199`、`myTopics.ts:107-110`)都**手动传 `where:{accountId}`** → 运行中不泄漏。
3. **真正泄漏窗口**:任何未来 resolver 仿照 `stepData.ts` 模式省略 where、依赖 RLS 兜底 → DB 返回**所有账户**的收藏 → 跨账户泄漏。
→ **即:trending_favorites 当前靠应用层 where 手动守,不是 RLS;一次"按 stepData 模式重构"就会捅出跨租户漏洞**。

`activeAccountId===null` → `FORBIDDEN('no_active_account')` fail-fast(安全,:28-31)。事务 timeout=70000ms(容纳 LLM ~45s+retry)。

## 四、承重 / 软肋(软肋附"为什么接受")

| | 判定 | 说明 |
|---|---|---|
| 三道闸(context→中间件→RLS policy) | **承重** | 租户隔离的全部依赖;中心化单点可审计 |
| **trending_favorites 漏 RLS** | **真缺陷(不接受)** | 表承诺 per-account 却无 policy;阶段3 已立修复 AC-A,→ 阶段6 补 |
| **ContextAssembler._fetchStepData 用全局 `prisma` 脱离事务 RLS**(`ContextAssembler.ts:18,151-157`) | **软肋(双轨隐患)** | 它在 `accountIsolationMiddleware` 的事务**之外**执行(用模块级 prisma 非 ctx.prisma=tx)→ 不在 RLS 上下文内,靠自带的 `where:{accountId}` 应用层守。接受:where 接地正确、accountId 来源可信(ctx.activeAccountId);**但形成"router 走 RLS、ContextAssembler 走应用层 where"的双轨**,一致性靠人记 |
| set_config 依赖事务边界 | **可接受** | is_local=true 事务提交即清,防连接池泄漏;SSE 用专用连接不复用 |

## 五、根本张力与关键赌注

**赌注:把租户隔离全压在 DB RLS 这一层**。赌对:消除应用层 where 散落的泄漏点、中心审计、kernel 不可绕。赌错代价:① **新增表必须同步 `manual_rls.sql`——已证实会漏**(trending_favorites);② 全栈失效模式 = **Agent prompt 的 L2 段被填入别账户的孵化历史 = "拿到别人的大脑"**(比传统数据泄漏更隐蔽,污染的是生成语境);③ `quanan_app` 角色须在 DB 存在且有表权限(部署外部依赖,integration test `rls-isolation.test.ts:31-40` 显式建角色)。
