## 一、系统架构总览

```
┌─ 前端 Vite+React SPA (apps/web) ── 9步向导 + 14工具页 + 6模块 ─┐
│   useStepData: LS先写 + tRPC异步写 · X-Trace-Id注入            │
└────────────────────────── tRPC (splitLink: batch/SSE) ─────────┘
                                  │
┌─ tRPC 路由层 (apps/api/src/trpc) ──────────────────────────────┐
│  traceMiddleware → accountIsolationMiddleware(RLS事务激活)      │
│  appRouter / adminRouter ; stepData.save 统一入口              │
└──────────── findStepAgent(stepKey) 规则派单 ───────────────────┘
                                  │
┌─ Specialist 执行层 (13 registry) ── BaseSpecialist.execute() ──┐
│  ①校验 →②ContextAssembler.assemble →③invokeLLM →④safeParse →⑤costLog│
└──────┬──────────────────────┬─────────────────────┬───────────┘
       │ assemble()           │ complete()          │ persist
┌──────▼─────────┐  ┌─────────▼────────┐  ┌─────────▼──────────┐
│ ContextAssembler│ │  LLMGateway      │  │ Postgres+pgvector  │
│ 7路并行(5s timeout│ │ 三tier+熔断+限流  │  │ +RLS / Redis       │
│ allSettled降级)  │ │ +计费(唯一LLM入口)│  │                    │
└────────┬────────┘ └──────────────────┘  └────────────────────┘
   L1-L5记忆/RAG
                                  │ (异步,非请求内)
┌─ BullMQ Workers (Redis broker) ───────────────────────────────┐
│ evolution(反馈飞轮) · daily-task(cron) · trending-scraper      │
│ + embedding · imageGen(DALL-E) · fileParser · stt/tts(待删)    │
└────────────────────────────────────────────────────────────────┘
```

> **前后端分离缝（R-22）**：前端类型 ← `@quanan/clients/router-types`（codegen shadow）+ `@quanan/schemas`（zod 契约），**不碰 `apps/api/src` 内部**（grep=0 已实测）；`packages/` = 类型契约分离层。详 §七·补。

**一句话架构**：QuanAn 是一个 **「确定性 Workflow 编排 + 规则派单 Specialist + DB 共享黑板上下文 + 单入口 LLMGateway + BullMQ 异步飞轮」** 的多租户 SaaS；核心同步数据流 = `用户提交 → tRPC(RLS) → findStepAgent → ContextAssembler 7路装配 → LLMGateway → 持久化 → 响应`。

## 二、组件清单与职责边界

| 组件 | 职责 | 边界（不负责） | 关键决策来源 |
|---|---|---|---|
| **前端 SPA**（`apps/web`） | 9步向导/工具页渲染；`useStepData` LS先写+tRPC异步写；X-Trace-Id 注入 | 不碰 DB/Redis/LLM；不含业务逻辑；不校 RLS | — |
| **tRPC 路由层**（`trpc/`） | `/trpc`+`/trpc/admin`；traceMiddleware；**accountIsolationMiddleware 开事务激活 RLS**；`stepData.save` 统一入口 | 不直接调 LLM SDK；不实现业务（派给 Specialist） | ADR-01 编排 / ADR-05 多租户 |
| **Specialist 执行层**（`specialists/`，13个） | `BaseSpecialist.execute()` 模板 5 步；按 stepKey 硬路由 | 不 import LLM SDK（经 LLMGateway，R-1 红线）；不管事务 | ADR-01 |
| **ContextAssembler**（公共层） | 7 路并行装配 systemPrompt（L2/L4/RAG/常量/prompt/dbConst）；piiMask+`<user_input>`定界 | 只读不写 DB；不调 LLM | ADR-04 记忆 |
| **LLMGateway**（`workers/llm-gateway`） | **唯一 LLM 入口**；三 tier 路由；限流→primary→fallback→模板三级降级；计费 | 不管业务；不管事务 | ADR-01 / 横切成本 |
| **BullMQ Workers**（`workers/`） | evolution/daily-task/trending-scraper + embedding/imageGen/fileParser；异步飞轮 | 不在请求链内；worker 直连 prisma（显式带 accountId，无 RLS ctx） | ADR-04 / 重构③ |
| **记忆五层**（`memory/`） | L1 Redis缓冲(语音,待删)/L2 stepData/L3 histories/L4 profile/L5 trending(stub) | L3=Knowledge RAG（非个人记忆，ADR-04 已正名） | ADR-04 |
| **存储层** | Postgres(RLS)+pgvector(检索)+Redis(L1/限流/BullMQ broker)+S3(Asset) | 前端不直访；pgvector 复用 PG 主库（ADR-012） | ADR-05 |

## 三、组件交互与接口契约

| 交互 | 输入契约 | 输出契约 | 失败/降级契约 |
|---|---|---|---|
| 前端 → `stepData.save` | `{stepKey:string≤64, inputs:record}` | `{ok, data:{stepKey,inputs,result,isFallback,version,updatedAt}}` | tRPC error → 前端 LS 已先写兜底 |
| tRPC → Specialist `execute()` | `{accountId, mode?, userInput, traceId, stepKey}` | `{result, isFallback, durationMs, tokensUsed, modelUsed, traceId}` | 可恢复错(5xx/429/timeout/schema二次失败)+有 fallbackTemplate → isFallback=true；无 → re-throw |
| Specialist → `ContextAssembler.assemble()` | `{agentId, accountId, mode?, userInput, needRag?, needLayers?}` | `{systemPrompt, userPrompt(PII-masked), tools, evolutionInsight?, metadata:{contextTokens,layersUsed,ragHits}}` | 7 路任一 5s 超时 → allSettled 降级空段 |
| Specialist → `LLMGateway.complete()` | `{model_tier, systemPrompt, userPrompt, responseFormat, metadata:{trace_id,agentId,accountId,userId}, timeout_ms?}` | `{content, tokens, model, duration_ms, trace_id, fallback?}` | 限流→primary retry1→fallback model→模板兜底 |
| Worker ← BullMQ Job | evolution `{accountId,triggerType}` / daily-task `{accountId,scheduledDate}` / trending `{sourcePlatform,sourceItemId,sourceUrl,rawContent}` | 各自 persist（upsert 幂等） | attempts=3 exponential 5s；jobId 去重 |

**关键契约约束**：① 所有 Specialist 经 `BaseSpecialist.invokeLLM`→`llmGateway.complete()` 单入口（无直接 SDK，R-1 红线）；② RLS 上下文仅在 tRPC 事务内有效，Worker 层无 RLS ctx 需**显式带 accountId**；③ step5/step7 走 `saveStream`(SSE) 用 connection-level `SET`（非事务 `SET LOCAL`，因 SSE generator 跨事务边界）。

## 五、数据流与状态管理

**主同步链**（9 步向导一次请求，以 step1→PositioningAgent 为例）：
`前端 useStepData(LS先写) → tRPC stepData.save → traceMiddleware → accountIsolationMiddleware(事务 SET LOCAL ROLE + set_config account_id 激活RLS) → prisma.stepData.upsert(version+1) → findStepAgent('step1')→PositioningAgent → BaseSpecialist.execute[①校验 ②ContextAssembler.assemble(7路并行) ③invokeLLM→LLMGateway.complete(限流→primary→fallback) ④outputSchema.safeParse(失败retry1) ⑤_writeCostLog] → persistStepAgentResult → afterPersist(step7写history) → 响应`

**异步链**（脱离请求，BullMQ）：① **Evolution 飞轮**：feedbackLog→`enqueueIfThresholdMet`(阈值[5,20,50,100]原子递增,jobId去重)→evolutionWorker(concurrency5)→EvolutionAgent→`$transaction([profile.update, insight.create])`→异常告警；② **DailyTask**：cron `0 0 * * *`Asia/Shanghai fan-out→dailyTaskWorker→DailyTaskAgent→dailyTask.upsert(幂等)；③ **TrendingScraper**：外部触发→worker(有 `stop_trending_scraper` kill-switch)→autoVerdict 合规扫描→写 `trendingReviewQueue`(禁直写主表)→audit_log。

**状态存储**：会话态=前端 LS + Redis L1(语音,待删)；持久态=Postgres(stepData/histories/evolution/cost_log)；多实例一致性=L4 evolutionProfile 走 DB 共享（原子事务防竞态），BullMQ jobId 去重防重放。

## 七·补、前后端分离边界〔2026-06-18 NFR〕

分离缝 = **tRPC 契约层**：前端 `apps/web` 经 `createTRPCReact<AppRouter>`（`packages/clients` 生成类型）+ `packages/schemas`（zod 契约）调后端，**不 import `apps/api/src` 内部模块**（grep=0 已实测）；前后端独立 build/部署（apps/web 仅需生成的 router types，不需后端源码）。本节把"运行时分离"显式升为**"类型契约分离边界"**——Harness **R-22** 强制（前端禁 import 后端内部）。承接 NFR-前后端分离 / 阶段2 前后端分离方案（保留 tRPC）。

**实现机制（shadow router-types · LOAD-BEARING）**：`AppRouter`/`AdminRouter` 类型由 `pnpm codegen:shadow`（`scripts/codegen-shadow.cjs`，`package.json:50`）从真后端 router **自动生成**为扁平自包含 shadow（`packages/clients/src/*.generated.ts`）；CI `ci.yml:34-46` **重生成 + git diff 防漂移**（生成物过期=job 红，提示跑 `codegen:shadow`）。**前端不可直接 import 真 `AppRouter`**——会因 api 的 `@/` 自引用串扰 TS2307 + AdminRouter 7 道中间件超 `createTRPCReact` TS 实例化上限（`trpc.ts:8-9`/`router-types.ts:6-13` LOAD-BEARING 注释）。**这个 codegen shadow 不是可删的债、是分离缝的工程化落地**——手写 shadow 曾漂移、被错误类型掩盖 `cost.aggregate`/`judgeScore` 等线上真 bug（`ci.yml:37`），故改自动生成 + CI 守门。这正是 R-22 要前端走 `packages/clients` 而非直接 import 后端的**防御型依据**。

## 八、风险、权衡与未决项

**对账债务结论（阶段3 清）**：
1. **RLS 表数**：权威 = `manual_rls.sql` **16 张** ENABLE RLS（非 DATA-MODEL 说的 12、非 CLAUDE 说的 18）。**文档漂移**：DATA-MODEL.md:22/2143/3571、CLAUDE.md:87、manual_rls.sql:14 注释均需改为 16 → **修复 AC-C：文档对齐，阶段6**。**真缺陷**：`trending_favorites` 漏 RLS（修复 AC-A）。
2. **Specialist 名册**：canonical = **13 step Specialist**（registry `_specialists`，含 Presentation/VoiceChat）+ **Evolution/DailyTask 2 个 L5 自治 Agent**（走 BullMQ 不进 registry=架构意图非缺陷，AGENTS 白名单豁免）= **15 个独立 Agent class**；AGENTS.md 锁定的「3 L5 自治」=VoiceChat（step+L5 双身份，已计入 13）+Evolution+DailyTask（故"3 L5自治"与"13 step"有 VoiceChat 重叠，独立类共 15）。**漂移**：`SPECIALIST_IDS`(schema) 缺 VoiceChat、含 Presentation；AGENTS.md 含 VoiceChat、缺 Presentation——两者互补不一致。**因 VoiceChat 阶段6 删除（重构⑦），此漂移随语音下线一并消解**（阶段6 统一名册）→ 不单独修，归阶段6。

**已知风险/缺陷**：
- ❌ **G节⛔3 API Key 明文落库**（修复 AC-B，阶段6 加密）——本阶段设计层已定加密方案，实装待阶段6/8 verify。
- ⚠️ **trending_favorites 跨租户漏 RLS**（修复 AC-A，阶段6）。
- ⚠️ **J节② RAG 质检门**：knowledge_chunk admin 受控写入（免开放质检门）；TrendingItem 开放摄取由 TrendingScraper `autoVerdict`+`reviewQueue` 三态雏形覆盖，完整五触发点 + chunk 入库注入过滤 → 修复 AC① + 阶段6。
- ⚠️ 阶段2 三修复 AC（注入特征过滤/成本护栏/PII 二次扫描）架构落点已定（见六），实装阶段6。
- ⚠️ **LLMGateway 熔断注释失实**：文件头注释称「5次5xx熔断」但代码仅 retry1 无 circuit breaker（A 节熔断为软项，注释过度声称）→ 阶段6 修注释或补熔断。
- ⚠️ L4 记忆无衰减（P2 债）；L1/L5 待清理（语音删/数据源）。

**关键权衡**：① 不引 Supervisor（牺牲动态编排灵活性，换 Workflow 确定性 + Solo 可维护）；② RAG 纯向量（牺牲精确匹配，换内容创作场景够用 + 简单）；③ knowledge_chunk 全局共享（牺牲租户隔离粒度，但知识本就公共、无泄露面——故 cross_2「向量库租户过滤」对 QuanAn 为 N/A）。

**未决项（后续阶段/用户决策）**：对标第三方 API 平台选型[未实测]（阶段2/3）；文件解析引擎选型（阶段3/重构②）；红队测评机构（阶段8）；容量压测（阶段9）。
