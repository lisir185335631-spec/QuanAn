# QuanAn 方案评审（阶段2 战略评审与方案选型）

> **项目**：QuanAn · AI IP 孵化平台（内容创作+营销增长复合，复刻 aiipznt.vip）·**存量重构**
> **画像**：P1=自定义(内容+营销) · P2=创作容错 · P3=公网云 · P4=Solo · P5=多模态 · P6=定时·事件·批量 · P7=有人工复核 · P8=低 · **P9=多租户=是** · **P10=多Agent协作** · P11=无特殊 · P12=混合
> **事实底座**：5 路 Sonnet 回源核 QuanAn 真实代码（file:line）+ 对照手册条目（编号），Opus 把关亲核门禁关键条目（2026-06-18）。
> **方法**：存量重构的阶段2 = ① 逐簇复审现有方案是否成立（对照真代码）② 重构 8 主题战略影响落成 ADR ③ 风险识别（过 02 清单 D 节 4⛔）。

## 一、范围与方法

QuanAn 是**存量重构**——编排/RAG/记忆/多租户等核心方案代码里已有。本阶段不是从零选型，而是：
1. **复审现有方案**：对照手册条目核现有范式按画像是否成立、有无过度设计或缺口。
2. **重构 ADR**：把 `重构.txt` 8 主题的战略影响落成方案决策（详见 §四汇总表）。
3. **风险识别**：过 02 清单 D 节硬门禁，识别上线前必修风险。

**结论速览**：现有编排/RAG/记忆/多租户方案**整体成立、无需推翻**；重构 6 条（①②③④⑤⑥）有数据模型/模块改动需阶段2 ADR，2 条（⑦删除/⑧维持）落阶段6；**成本预算护栏（D9.3 C 层）+ 注入特征过滤（0.7 A 层）+ specialist PII 二次扫描 + 输出合规（12.3）是已识别的实现债务，须阶段2 立修复 AC、阶段6/8 落地**。

## 二、方案 ADR 组

### ADR-01 编排范式 — 维持，不引 Supervisor
- **决策**：维持「L4 tRPC `stepData.save` 统一入口 + `registry.ts` `findStepAgent` 规则派单 + BullMQ 异步后台」范式，不引入中央 Supervisor Agent。
- **现状**：9 步向导无 per-step router，全经 `stepData.save`→`STEP_AGENT_REGISTRY`（`registry.ts:103-170`，8 条静态映射）→`agent.execute()`；agent 间不直接通信，经 DB（`StepData`）共享；ContextAssembler 7 路并行拉 DB 注入 systemPrompt（`ContextAssembler.ts:53-129`）；LLMGateway 单入口（`workers/llm-gateway/index.ts:83-86`，三 tier+熔断）。
- **手册依据**：4.10（流程固定的标准产品流水线→确定性 Workflow 而非 Supervisor）；3.4（9 步纯静态→静态工作流，不满足引 Temporal 三维判据）；2.3（`findStepAgent` 纯规则路由，优于 LLM 自主派单）；4.5（ContextAssembler=共享黑板/DB 变体）；D3.3（registry 模式=单→多 Agent 最佳预留，加 Agent 只加一条目）。
- **画像核**：P10「多Agent」实为**并发用户各自独立跑 Specialist**（同一请求内无 Agent handoff/结果合并）→ 按 4.10 更接近 P4 Solo 用多职能域 Specialist，**不需 Supervisor**，成立。P2 创作容错经 `BaseSpecialist` fallback 三级兜住。
- **重构影响**：删语音（⑦）= `registry._specialists` 删 voiceChat（`registry.ts:69`）+ `_app.ts` 删 voiceChatRouter（:32/:62）+ `history.ts:44` 清引用；`STEP_AGENT_REGISTRY` 无需动（VoiceChat 本就不在其中）。
- **风险**：见 §三 R-编排（L2 token 膨胀 / step5·7 双持久化 / Specialist 全局并发无硬上限）。

### ADR-02 RAG 方案 — 维持纯向量，收口 TrendingItem 向量
- **决策**：维持 pgvector 纯向量语义检索（knowledge_chunk：case/formula/element 三类），不引混合检索（BM25）；阶段2 RAG 工作=等重构③数据源落地后收口 `trending_items.content_embedding` 写入路径。
- **现状**：`workers/rag/retrieve.ts` 纯向量 cosine `<=>` top-K + dev ILIKE fallback；`OpenAIEmbeddingWorker`（`workers/embedding/openai-embedding.ts`，text-embedding-3-small/1536 维，D-038 不走 LLMGateway）；ContextAssembler `_fetchRag` 按 agentId 联动检索策略（`ContextAssembler.ts:169`）；**TrendingItem 向量空转**（索引已建 IVFFlat 但 `l5-trending.ts` stub `return []`，`_fetchRag` 不查 trending）。
- **手册依据**：7.4a/7.4b（内容创作非精确匹配→纯向量足够，tsvector 混合路是可选升级）；7.13（`_fetchRag` 已按 agentId 做意图→检索策略联动）；F6（案例/公式/元素=稳定知识走 RAG、TrendingItem=高频更新必须 RAG 不微调，判断正确）。
- **重构影响**：RAG 本体无需改；数据管线上游待重构③第三方 API 落地→TrendingScraper Worker 管道内增 EmbeddingWorker 调用写 `content_embedding` + `_fetchRag` 扩 trending 检索路（TopicAgent 按 industry+platform 召回对标爆款）。

### ADR-03 数据获取 — 自爬否决（已是数据模型红线）→ 合规第三方 API
- **决策**：**自建爬虫已被否决并写入数据模型红线**；对标数据走合规第三方授权 API（新榜/蝉妈妈/飞瓜，**[未实测]·[待选型]**），由 Cron 定期拉取→投递 `trendingScraperQueue`→已有 Worker 合规审核入库。
- **现状**：`vendor` 字段红线（`DATA-MODEL.md:276-278/319`：枚举 `'xinbang'|'cmm'|...`，**严禁 `'self_crawler'`，R-17 grep 检测**）；TrendingScraper BullMQ 框架已落地（`workers/trending-scraper/queue.ts/worker.ts`：autoVerdict 合规扫描→写 `trendingReviewQueue` 不直写主表→admin 审核）；**数据注入端空缺**（`l5-trending.ts:17-23` stub，TODO PRD-9）。
- **手册依据**：D12.2（第三方 API 接入=REST 对接+字段映射层）；D12.3（结果回写对账）。
- **权衡（业务可行性三角）**：自建爬虫=违反平台 ToS/法律风险高+反爬维护成本极高 → 否决；第三方授权 API=合规+SLA 稳定，代价=API 套餐费用[待评估]。
- **重构③④落地**：Cron→第三方 API Client→字段映射→`TrendingScraperJobPayload`→已有 Worker。**实现层须守**：永不对 `vendor='self_crawler'` create/upsert；trending 路径 `grep puppeteer/playwright-core` 须 0 命中（§三③ AC1）。
- **未决**：① 选哪家 API [待选型]（阶段2/3）；② `authorFollowers`+`industry` 过滤能力（低粉阈值用户自填）；③ Cron 频率（建议 4h，[未实测]）。

### ADR-04 记忆方案 — 五层记忆保留，三处明确决策
- **决策**：五层记忆（L1 Buffer/L2 Core/**L3 全文档历史**/L4 Profile/L5 RAG pgvector）+ 反馈飞轮整体保留；**明确三处**：① L1 重构⑦删语音后空置→同步标废弃；② **L3 = 全文档历史（`histories` 表，只写不读，非承重，非外部知识检索）；外部知识检索 = L5 RAG pgvector**（已与 `03-记忆.md` 中 L3 定义对齐，旧"L3 Recall/Knowledge RAG"命名已不存在，本 ADR 使用现行 03-记忆.md 的 L3 定义）；③ L5 stub→等数据源（ADR-03）。
- **现状**：L2 全量注入 stepData（`ContextAssembler._fetchStepData`）；L4 Profile `latestInsight` JSON 热缓存全量注入（`l4-profile.ts`）；反馈飞轮完整（`lib/evolution/trigger.ts:13` 阈值 [5,20,50,100] + `EvolutionAgent.ts:182-387` 原子事务 + BullMQ 异步）；L1 Redis TTL 1800s 专为 VoiceChat（`l1-buffer.ts`）。
- **手册依据**：8.3（写入异步不卡主链路，符合）；8.5（L2 量小全量注入合理，P8 低规模单账号 ~9 条；**L5 RAG** 外部知识检索量小、IVFFlat 在 <10K 条轻度过度设计但有 ILIKE fallback 兜底）；8.10（陪伴型需衰减，QuanAn L4 无 Ebbinghaus 衰减→**P2 技术债**，P8 规模可接受、长期补）。
- **风险/债务**：L4 衰减缺失（旧偏好永驻，长期噪声累积，P2 债）；L4 BullMQ 崩溃重放幂等待核（原子事务防重复但无显式 cursor）→阶段6 核。

### ADR-05 多租户隔离 — 三道闸，维持
- **决策**：维持「tRPC 事务中间件 → Postgres RLS → pgvector/Redis 命名空间」三道闸，以 RLS 为数据兜底。
- **现状**：tRPC 主路径（`middleware/account-isolation.ts:56-65`：事务内 `SET LOCAL ROLE quanan_app`+`set_config('app.current_account_id')`，无 activeAccountId→FORBIDDEN）；RLS 16 张业务表（`manual_rls.sql`，子表 `account_id=current_setting(...)` 隔离；**trending_favorites 漏 RLS 待补 · R-21/US-006**）；admin **28 表**无 RLS 改由 `adminRLS` 中间件守（`middleware/admin/adminRLS.ts`，`manual_admin_rls.sql` 实数：16+6+4+2=28）；TD-019 后补 `where:{id,accountId}` 显式双闸（history/videoProduction/aiVideo）；pgvector 检索按 accountId 过滤。
- **手册依据**：D11.1（三道权限 A 统一模型+B 上下文贯穿+C DB 层强制过滤，QuanAn 落地此模式）；D13 多租户面（公网云 SaaS，非私有化，D13.1 私有化条文不适用）。
- **风险/取舍**：admin 表无 RLS=单点依赖应用层中间件（有测试兜底，接受）；SSE 用 `SET`（非 `SET LOCAL`，事务生命周期不兼容，连接专用接受）；trending 全局共享表（对标数据非私有，有意设计）。
- **补充风险（阶段6 登记）**：
  - **G15 · ContextAssembler._fetchStepData 双轨**：任何重构若按 `stepData.ts` 模式省略 `where:{accountId}` 则脱离事务 RLS 跨租户泄漏（`_fetchStepData` 走 `prisma.$transaction` 内部，`SET LOCAL` 在事务提交后失效，`accountId where` 子句是唯一保证）→ US-002 AC-6 守线；阶段6 重构中禁止移除。
  - **R-21 · trending_favorites 漏 RLS**：`prisma/schema.prisma` 中 `trending_favorites` 含 `account_id` 但 `manual_rls.sql` 未 ENABLE RLS，R-21 audit 检测会抓出此表 → US-006（P0）必修；补齐后 R-21 PASS。

### ADR-06 变现/计费 — D14.3 配额已落，D14.7/D14.8 不启动
- **决策**：变现路径（Step4b）=内容创作 AI 规划输出，与计费无关（主题⑧维持原样）；平台计费仅 D14.3 配额层（UserQuota/CostLog）已落地，阶段2 **不新增对外计费**，D14.7 任务市场/D14.8 对外计费均不启动。
- **现状**：MonetizationAgent 后端已实现（`MonetizationAgent.ts:94-118`，reasoning tier）；Step4b 前端 ⚠️脱节（`Step4b.tsx:387` `generateMockResult()` mock 壳）；UserQuota 原子扣减已实现（`services/quota/userQuota.service.ts:39-43`）；支付网关/订阅升级**零实现**（schema 预留 plan/lastUpgradedAt）。
- **手册依据**：D14.7（任务市场=商家→创作者奖励体系，与 QuanAn AI 规划输出场景不匹配，不启动）；D14.8（P4 Solo+无对外 SaaS 收费→可不实施，D14.3 配额够用）。
- **风险**：Step4b 前端脱节=**实现层债务归阶段6/7**；阶段2 ADR 仅记录"不加新计费模式 + MonetizationAgent 接线范围"。

### 前后端分离方案〔2026-06-18 NFR 增补〕
- **决策**：分离缝 = **保留 tRPC 类型安全契约层**（monorepo `packages/schemas`+`packages/clients` 共享），**不换 REST/OpenAPI**。
- **依据**：QuanAn 无非 TS 客户端（mobile App / 对外开放 API）需求 → tRPC 端到端类型安全是优势；前后端独立部署（apps/web+apps/api）已具，契约层（packages）即分离边界。承接 1-需求规格 §九 NFR-前后端分离，Harness R-22 强制。
- **取舍**：保 tRPC 牺牲"跨语言/独立版本"换"端到端类型安全+零契约漂移"；若未来要 mobile/对外 API 再回溯阶段2 评估 REST/BFF。
